**Drift risks (same kind of data, different source):**
1. **Pulse/Register vs Central.** Pulse and Register read the baked `const DATA` (sheet transcription); Central reads the live SQLite DB. Same campaigns/pacing/margin, two independent stores — they diverge whenever one updates and the other does not.
2. **Two formula engines.** Pulse/Register (and `reconcile.js`) compute derived fields with `src/derive.js`; Central computes them with `src/central/calc.js`. They differ on at least: CPM Performance (`derive.js` uses `clientSpent/impressions`, L98-103; `calc.js` uses `mediaSpend/impressions`, L142-145), % budget spent base (`derive.js` `totalBudget`; `calc.js` `effectiveBudget` = budgetGross-first, L101/L111), and pacing tolerance (`the-grid.html` `PACE_BAND=0.15` L692 / `derive.js` default 0.10 L150; `calc.js` uses 1.05/0.95 bands L164-166). Same field names, different numbers possible.
3. **Executive is a third source.** Exec KPIs come from each client's GCS `data.json`, independent of both `const DATA` and the Central DB (different metric — client outcome KPIs, not pacing).

## 4. The CONFIG / API / DERIVED model

Origin tags are defined in `src/central/calc.js:13-16` and the `Campaign` typedef L30-68. Write governance (the enforcement) is in `src/brain/db.js:85-101`.

**CONFIG (typed / plan-sourced):** `agency`, `client`, `currency`, `jobNumber`, `name`, `objective`, `channel`, `managedBy`, `status`, `startDate`, `endDate`, `platformMargin`, `adServing`/`adServingCost`, `forecastCpm`, `keyKpi`, `kpiTarget`, `budgetGross`, `totalBudget`, `campaignLink`, `nextReportingDue`, `notes`, `spendMult`, `kpiPerformance`.
- Plan-commit writable set: `CENTRAL_PLAN_FIELDS` (`db.js:87-89`).
- Inline-edit writable set: `CENTRAL_EDIT_FIELDS` (`db.js:94-96`).
- Note: `kpiPerformance` was reclassified from DERIVED to editable CONFIG — "the sheet's KPI Performance is hand-typed text (calc.js's passthrough was a never-implemented TODO)" (`db.js:98-99`).

**API (synced from platforms):** `impressions`, `mediaSpend`, `clientSpend`.
- On sync: `clientSpend = mediaSpend × spendMult` when `spendMult` is set; when unset, `clientSpend` is left untouched (never written as `mediaSpend`) and the row shows an unbilled-basis badge (per README "Sync" section).

**DERIVED (computed, never writable):** `campaignMargin`, `cpmPerformance`, `budgetRemaining`, `pctBudgetSpent`, `pctFlightElapsed`, `pacingStatus`, `marginDelta`, `marginBand`, `health` (`CENTRAL_DERIVED_FIELDS`, `db.js:100-101`; rejected by `centralFieldAllowed()` L202-206 and `createCampaign()` L312). `adServingCost` is also computed in `calc.js`.

**Where the formulas live:** `src/derive.js` (Pulse/Register + `reconcile.js`) and `src/central/calc.js` (Central). Two separate engines (see §2 drift #2).

**TTD Platform Margin vs Campaign Margin — the code does NOT branch by channel.**
Every channel (including Trade Desk) uses the SAME `campaignMargin` formula. Platform Margin is a manual CONFIG input, used only to compute `marginDelta`/`marginBand` (realized-vs-set), never as a per-channel margin. Relevant lines:

- `src/derive.js:90-96` — `campaignMargin = (V - U - M) / V` (clientSpent − mediaSpend − adservingCost) ÷ clientSpent.
- `src/derive.js:221` — "K Platform Margin = MANUAL INPUT (no formula) — passthrough, not derived".
- `src/central/calc.js:132-138` — `campaignMargin = (clientSpend - mediaSpend - adServingCost) / clientSpend`.
- `src/central/calc.js:176-181` — `marginDelta = campaignMargin - platformMargin` (the only place platformMargin enters math).
- `src/central/calc.js:42` — `platformMargin` typedef tag `[CONFIG]`.

There is no `if (channel === 'Trade Desk')` in either margin path. The premise "TTD uses Platform Margin, other channels use Campaign Margin" is not implemented in the code as written.

## 5. The media-plan reader — current state

- **Lives in:** `src/central/plan-reader.js` (server-side extraction); UI dropzone/review in `src/central/plan-panel.js`; routes `POST /api/central/plan/upload`, `/api/central/plan/:id/commit`, `/api/central/plan/:id/discard` (`server.js:131-133`).
- **Formats accepted:** XLSX/XLS/CSV (native via SheetJS) and PDF/DOCX/PPTX (via `src/brain/parser.js` text extraction). `plan-panel.js` accept list and `plan-reader.js:39` sheet-name priority `/plan|media|brief|launch|budget/i`.
- **Fields extracted:** `TARGET_FIELDS` (`plan-reader.js:26-28`) = `jobNumber, client, name, objective, channel, managedBy, startDate, endDate, platformMargin, adServingCost, forecastCpm, keyKpi, kpiTarget, budgetGross, totalBudget, spendMult, notes` — all CONFIG.
- **Provenance + confidence:** Yes. Each extracted field carries `{value, sheet, cellRef|page, confidence}`; low-confidence flagged in the review panel (`plan-panel.js:154` `ct-lowdot` "low confidence — please verify"). With `ANTHROPIC_API_KEY` it uses Claude; without, a deterministic header-keyword heuristic tagging everything `confidence:'low'` (`plan-reader.js:22`, mode selection L333).
- **Auto-write or stage?** **STAGES for human review.** `plan-reader.js` header (L4-6): "It NEVER writes a campaign row — it produces a draft that a human reviews and commits." Upload → PENDING draft `{fields, candidates}`; commit route writes only user-confirmed values, rejects unacknowledged overwrites and derived fields, and can create a new row (`sourceOfRecord:'plan'`).

## 6. BQ reconciliation / coverage

- **Config file:** `config/central-clients.json`. Each client has `validated` (bool), `source` (`view` / `raw` / `none`), and a `map`.
- **Validated today:** **Schneider only** (`central-clients.json:6`, `"validated": true`, Mode A `view` = `client_schneider.pm_delivery`).
- **Not validated (`validated:false`):** HireRight (source `raw`), Cloudflare, STT, PropTrack, MongoDB, City Perfume, ResetData, VMCH, The Little Marionette, QTopia, Ad Assembly, Gateway, Caltex, Next Smile Australia, Bell Shakespeare. Most have empty `map:[]`. These render sheet/import values, not reconciled BQ metrics.
- **Reconcile flow:** `GET /api/central/reconcile/:client` (`server.js:130`) → BQ campaign-name list + fuzzy-scored SUGGESTIONS against Central rows (never written). `POST /api/central/reconcile/:client/approve` (`server.js:129`) → writes the human-approved pairs into `central-clients.json` `map` and sets `validated:true` (`server.js:565` returns `validated: spec.validated`). UI entry = "Map client" button (`render-central.js:342`).
- **What "validated" means in code:** the `validated:true` flag gates the sync — `scripts/central_sync.py` / `centralSync` query BQ ONLY for validated clients; non-validated clients are returned under `skippedClients`. Matching at sync time uses the explicit `map` only (no fuzzy).

## 7. Scheduling / freshness

| `scripts/central_sync.py` | `POST /api/central/sync` ("Sync now" button); optional interval | Interval **OFF by default** | `CENTRAL_AUTOSYNC_MIN` default **0 = off** (`server.js:417`, interval L627-630). Central live metrics do not auto-refresh unless the env var is set; otherwise only on manual click. |
| `src/brain/bq-writer.js` (V3.5) | On historical commit | Event-driven | Best-effort; not scheduled. |

Other freshness notes: `config/exec-kpis.json` is a build-time snapshot (freshness = last manual/interval build). `const DATA`'s `dataThrough`/LIVE tags come only from a `build_grid_data.py` run.
